# Kairove Source Scout and Harvest Plan

## 0. Purpose

This document defines how Kairove discovers, collects, records, and prepares external sources for trend and format mining.

The Source Scout and Harvester are Kairove's eyes and ears. If this layer is weak, Format Miner will not know what people are actually making.

This layer must answer:

- what videos are currently hot;
- what scripts are repeated;
- what music or sound is spreading;
- what AI styles are being used;
- what comments indicate audience appetite or fatigue;
- which sources can be directly used, referenced, or only analyzed;
- what API, login, or platform setup is missing.

---

## 1. Scope

This plan covers:

1. Source discovery;
2. platform-specific scouts;
3. query planning;
4. candidate normalization;
5. candidate ranking;
6. harvesting strategy;
7. metadata, screenshots, comments, transcript, audio reference collection;
8. source and asset manifests;
9. usage policy tagging;
10. failure handling;
11. handoff to Video Understanding Engine.

It does not cover deep format mining. That starts after understanding reports are produced.

---

## 2. Concepts

### 2.1 SourceCandidate

A lightweight item discovered before full harvest.

Examples:

- video search result;
- trending video;
- topic page;
- music page;
- wiki page;
- creator profile;
- manual URL.

### 2.2 Source

A harvested research object with manifest, provenance, local files, and usage policy.

### 2.2A EvidenceObservation

A record of what Kairove actually saw or obtained from a source before any understanding claim is made.

It separates:

```text
observed evidence:
  title, description, tags, cover URL, cover file, public HTML page snapshot, page metadata, page screenshot, transcript, comments, frames, audio metadata when available

missing evidence:
  anything not collected or not technically/permission-wise available
```

Understanding reports must be generated from EvidenceObservations, not directly from SourceCandidates.

When policy allows limited reference-asset download, P1 may store source cover/thumbnail images as `analysis_only` research assets. Cover files support weak packaging observation, not claims about full video content, rhythm, story, audio, comments, or transcript.

When metadata collection is allowed, P1 may store a public HTML page snapshot under `raw_pages/`, extract basic meta tags, and keep a limited set of public visible text snippets. This is not a browser screenshot or logged-in platform view, and it must not be used to claim that the video itself, comments, transcript, or recommendation context was viewed.

When `research.browser_screenshot` is allowed and a browser runtime exists, P1 may store one unauthenticated public browser screenshot as `analysis_only` evidence. It also records basic image quality metadata such as dimensions, file size, nonblank score, and likely blank/single-color status. It must not use user cookies, logged-in sessions, platform bypasses, or scrolling/comment capture unless a later permission explicitly allows that work.

### 2.3 Asset

Any file or capability created from a source:

- video file;
- screenshot;
- cover;
- comment JSON;
- transcript;
- audio reference;
- downloaded image;
- webpage snapshot.

## 3. Source Types

Kairove should classify sources by type:

```text
official:
  game/anime official websites, official PVs, official accounts, official music, official images.

personal_creator:
  individual artists, individual video creators, fan creators, unofficial edits.

platform_user:
  normal platform user videos without clear official status.

wiki:
  public wiki, fandom pages, encyclopedic sources.

search_result:
  generic search result before source verification.

generated:
  AI-generated by Kairove or another known generation tool.

user_provided:
  files or links provided by the user.

unknown:
  source status cannot be determined.
```

Current user policy:

- official sources can default to `direct_use`;
- personal creator and unknown sources require review before final direct use;
- official-like or PV-like misleading risk is not blocking;
- all sources still require provenance recording.

---

## 4. Usage Policies

Every source and asset must have a usage policy:

```text
analysis_only:
  Can be used for trend, format, comment, or style analysis only.

reference_only:
  Can guide prompts or descriptions, but should not be directly placed in final video.

style_analysis:
  Can be used to extract style rules such as subtitle placement, cover layout, lighting, rhythm.

direct_use:
  Can be used directly in final assets under current policy.

generated:
  Generated by Kairove and can usually be used directly, with prompt/model provenance.

user_provided:
  Provided by user and usable according to user notes.

blocked:
  Must not be used.

unknown:
  Needs review or further classification.
```

## 5. Source Scout Overview

Source Scout does broad discovery.

It should not do expensive deep analysis. It should produce candidate sources for harvesting.

Pipeline:

```text
1. Intake
2. Query Planning
3. Platform Scout Execution
4. Candidate Normalization
5. Deduplication
6. Initial Relevance Ranking
7. Harvest Recommendation
8. Scout Report
```

## 6. Intake Modes

### 6.1 Auto Trend Patrol

Find broadly trending formats.

Inputs:

```json
{
  "mode": "auto_trend_patrol",
  "platforms": ["bilibili", "douyin", "xiaohongshu", "youtube"],
  "categories": ["ai_video", "meme", "horror", "anime_game", "music"],
  "time_window": "recent",
  "max_candidates": 500
}
```

### 6.2 Keyword Search

Find trends around keywords.

Inputs:

```json
{
  "mode": "keyword_search",
  "keywords": ["抽象短剧", "AI恐怖视频"],
  "platforms": ["bilibili", "douyin"],
  "max_candidates": 200
}
```

### 6.3 Manual Seed Links

Analyze specific user-provided links.

Inputs:

```json
{
  "mode": "manual_seed",
  "urls": ["..."],
  "intent": "analyze_and_reproduce"
}
```

### 6.4 Character / Fandom Research

Find official and wiki references for characters, CPs, worlds, and related videos.

Inputs:

```json
{
  "mode": "fandom_research",
  "world": "...",
  "characters": ["..."],
  "source_preference": ["official", "wiki", "platform_video"]
}
```

### 6.5 Audio Trend Research

Find videos using a specific music or sound pattern.

Inputs:

```json
{
  "mode": "audio_trend",
  "audio_ref": "...",
  "platforms": ["douyin", "bilibili"]
}
```

## 7. Query Planner

The Query Planner creates platform-specific search plans.

It should generate:

- direct keywords;
- synonym keywords;
- platform slang;
- fandom-specific terms;
- format-related phrases;
- music or lyric phrases;
- creator names if relevant.

Example:

```json
{
  "base_intent": "find abstract short drama trends",
  "queries": [
    {
      "platform": "bilibili",
      "query": "抽象 短剧 AI",
      "purpose": "general trend"
    },
    {
      "platform": "douyin",
      "query": "抽象短剧 文案",
      "purpose": "script repetition"
    }
  ]
}
```

## 8. Platform Scouts

### 8.1 BilibiliScout

Purpose:

- search videos;
- inspect hot/ranking pages if available;
- collect metadata;
- find repeated scripts, anime/game formats, AI videos, meme edits.

Candidate fields:

- title;
- URL;
- uploader;
- view count;
- danmaku/comment indicators if available;
- publish time;
- tags;
- category;
- cover.

Capability states:

```text
search: expected available
metadata: expected available
comments: may require extra handling
danmaku: later
download: optional / policy-dependent
```

### 8.2 DouyinScout

Purpose:

- find hot short video formats, songs, sound memes, short drama patterns.

Capability notes:

Douyin may require:

- browser automation;
- login;
- official API;
- manual seed links.

Capability states:

```text
search: may require browser/API
metadata: partial
comments: likely requires auth/browser
download: limited
```

### 8.3 XiaohongshuScout

Purpose:

- find cover/title packaging trends;
- note-like visual packaging;
- AI images/videos;
- fandom and lifestyle meme formats.

Capability notes:

- scraping may be fragile;
- browser assistance may be needed;
- manual review likely important.

### 8.4 YouTubeScout

Purpose:

- find Shorts trends;
- find AI video formats;
- find longform references;
- fetch metadata via API where possible.

Capability states:

```text
search: API or web
metadata: API possible
comments: API possible
metrics: partial
download: optional/reference
```

### 8.5 TikTokScout

Purpose:

- detect global short video trends.

Capability notes:

- region and login may affect access;
- may rely on browser/manual links.

### 8.6 WeiboScout

Purpose:

- detect hot topics, fandom discussion, official posts, platform sentiment.

### 8.7 SearchEngineScout

Purpose:

- find official pages;
- find wiki pages;
- find discussions;
- find trend reports;
- find tool references.

### 8.8 WikiScout

Purpose:

- find character references;
- find world information;
- find relationships;
- find official names and visual descriptions.

Preferred sources:

- official sites;
- game wiki;
- anime wiki;
- fandom pages;
- public encyclopedic pages.

### 8.9 ManualSeedScout

Purpose:

- ingest user-provided links or files;
- preserve user notes;
- skip broad search if user wants direct analysis.

## 10. Deduplication

Deduplicate by:

- exact URL;
- canonical URL;
- platform item ID;
- title similarity;
- cover similarity;
- audio similarity later;
- video hash later.

Dedup result:

```json
{
  "kept_candidate": "cand_src_000001",
  "duplicates": ["cand_src_000009"],
  "reason": "same platform item id"
}
```

---

## 11. Initial Relevance Ranking

Ranking is not final Trend Analyst scoring.

It only decides what to harvest first.

Factors:

- relevance to query;
- platform metrics;
- recency;
- official source priority if doing character or asset research;
- potential format repetition;
- manual seed priority;
- diversity across platforms.

Output:

```json
{
  "candidate_source_id": "cand_src_000001",
  "harvest_priority": 8.4,
  "reason": [
    "high relevance",
    "recent",
    "similar titles appear repeatedly"
  ]
}
```

## 12. Harvest Planning

Before harvesting, Kairove should decide what to collect.

Harvest levels:

```text
metadata_only:
  title, URL, author, metrics, publish time.

snapshot:
  metadata plus page snapshot and cover.

analysis_pack:
  metadata, cover, screenshots, comments, transcript if available, audio reference.

full_reference:
  video/audio files if allowed and technically possible.
```

Harvest plan example:

```json
{
  "candidate_source_id": "cand_src_000001",
  "harvest_level": "analysis_pack",
  "collect": [
    "metadata",
    "cover",
    "screenshots",
    "comments_sample",
    "transcript",
    "audio_ref"
  ],
  "reason": "high format mining value"
}
```

## 13. Harvester Components

### 13.1 MetadataHarvester

Collects:

- title;
- description;
- tags;
- author;
- publish time;
- metrics;
- platform item ID;
- source type guess.

### 13.2 PageSnapshotHarvester

Saves:

- HTML if available;
- screenshot;
- text extraction;
- timestamp.

### 13.3 VideoHarvester

Saves video if allowed and possible.

If not possible, records:

- failure reason;
- whether browser capture is possible;
- whether manual file is needed.

### 13.4 ScreenshotHarvester

Extracts:

- cover;
- sampled frames;
- key screenshots;
- visual references.

### 13.5 CommentHarvester

Collects:

- top comments;
- recent comments;
- high-like comments;
- comment count if available;
- sentiment clues.

### 13.6 TranscriptHarvester

Collects:

- platform captions if available;
- OCR text later;
- ASR transcript later.

### 13.7 AudioReferenceHarvester

Collects:

- music name if available;
- audio ID if platform provides it;
- audio segment if allowed;
- audio reference metadata.

### 13.8 ManifestWriter

Writes source and asset manifests.

## 16. Usage Policy Assignment

Policy assignment should follow source type and user config.

Initial rules:

```text
official + image/video/music/sfx:
  direct_use

personal_creator:
  reference_only or analysis_only until user approval

platform_user:
  analysis_only by default unless source classified official

wiki:
  direct_use if official or user policy allows; otherwise reference_only

unknown:
  unknown / ask_user

generated:
  generated / direct_use

user_provided:
  user_provided / direct_use unless user notes say otherwise
```

## 17. Official Source Detection

Official detection should use multiple signals:

- official website domain;
- verified account;
- platform official marker;
- known official account list in Knowledge Base;
- URL pattern;
- user-provided confirmation;
- source page content.

Output:

```json
{
  "source_type_guess": "official",
  "confidence": 0.86,
  "evidence": [
    "verified account",
    "matches known official account list"
  ],
  "needs_review": false
}
```

If confidence is low, mark as unknown or ask user.

---

## 18. Personal Creator Detection

Signals:

- personal artist page;
- username not in official list;
- fan art tags;
- watermark;
- caption indicating fan creation;
- source domain for art platforms.

Personal creator assets should not be direct-use without review.

## 22. Handoff to Video Understanding

Harvester should pass sources to Video Understanding only when:

- source manifest exists;
- at least metadata exists;
- files or references are recorded;
- usage policy assigned;
- harvest status known.
- EvidenceObservation exists and explicitly records available and missing inputs.

Handoff item:

```json
{
  "source_id": "src_000001",
  "available_inputs": [
    "metadata",
    "cover",
    "screenshots",
    "comments"
  ],
  "missing_inputs": [
    "raw_video",
    "transcript"
  ],
  "understanding_mode": "partial"
}
```

If only metadata is available, the handoff remains `partial` and Video Understanding must not infer full visual rhythm, audio, comments, transcript, or exact plot beats.

## 23. Review Items Created by This Layer

This layer may create review items for:

2. personal creator asset direct-use request;
3. official account confidence low;

5. manual video file needed;
6. source blocked but user may override;

## 26. Minimum Acceptance Criteria

This layer is ready when Kairove can:

1. Start a scout run from a keyword or manual URL.
2. Produce normalized source candidates.
3. Deduplicate candidates.
4. Rank candidates for harvesting.
5. Harvest metadata and at least one local evidence file or snapshot where possible.
6. Write source manifest.
7. Write asset manifest.
8. Assign usage policy according to user rules.
9. Record failures explicitly.
10. Create review items for uncertain sources.
11. Handoff sources to Video Understanding.

---

## 27. Implementation Notes

1. Do not fake platform APIs.
2. If a platform requires login/API, record a tool setup need.
3. Manual seed links should be supported early.
4. Search engine and wiki discovery are important because many character and official assets can be found there.
5. Official source classification should be confidence-based.
6. Metadata-only harvest is acceptable if full harvest is blocked.
7. Every collected file needs provenance.
8. Avoid mixing research assets with local assets and generated assets.
---

## 27. Handoff to Asset Resolver

Source Scout and Harvester do not decide final production usage alone.

They hand source evidence to Asset Resolver with enough policy and provenance data to decide whether an asset can be used, referenced, generated from, or needs review.

Handoff package:

```json
{
  "source_id": "src_000001",
  "candidate_source_id": "cand_src_000001",
  "source_type": "official | personal_creator | platform_user | wiki | unknown",
  "content_type": "video | image | audio | page | comment_thread",
  "usage_policy_guess": "direct_use | reference_only | style_analysis | analysis_only | unknown",
  "review_status": "not_required | pending | approved | rejected",
  "local_files": [],
  "metadata_path": "...",
  "manifest_path": "...",
  "policy_evidence": [],
  "recommended_asset_roles": ["character_reference", "audio_reference", "cover_style"]
}
```

Rules:

- Source Scout may guess usage policy, but Asset Resolver confirms job-specific use.
- Official assets can be marked direct-use under current user policy, but provenance is still required.
- Personal creator and unknown assets must become ReviewItems before direct use.
- Missing download/API/login capability becomes ToolSetupItem.
- If only analysis/reference is allowed, final generation must not embed the asset directly.
